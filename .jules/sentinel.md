## 2025-03-05 - [Critical] SQL Injection in SQLite PRAGMA table_info via Unescaped Identifier
**Vulnerability:** Found an unescaped identifier (`table_name`) formatted directly into a `PRAGMA table_info` query (`f"PRAGMA table_info({table_name})"`) in `db_manager.py`'s `get_table_info` function.
**Learning:** SQLite's `PRAGMA` statements do not support standard prepared statement query parameterization. This often leads to developers falling back to string formatting, which introduces SQL injection vulnerabilities if the identifier is attacker-controlled.
**Prevention:** Since parameterized identifiers are not supported in `PRAGMA`, dynamically constructed queries using identifiers must be escaped manually. This is done by doubling interior double quotes (e.g. `"` -> `""`) and wrapping the entire identifier in double quotes (e.g., `f'"{table_name.replace('"', '""')}"'`) to ensure it's safely treated as a schema identifier and not executable SQL.

## 2025-03-05 - [Critical] Insecure Deserialization via pickle in CacheManager
**Vulnerability:** Found `pickle.load()` being used to deserialize cache files (`*.pkl`) from a world-writable directory (e.g., `/tmp/codomyrmex_cache`). This is a critical Remote Code Execution (RCE) vulnerability.
**Learning:** Storing cached object states using `pickle` in predictable, shared locations like `/tmp` allows local attackers to inject malicious code. Because `pickle.load()` can execute arbitrary code during the instantiation of deserialized objects, anyone with write access to the cache directory can achieve privilege escalation or arbitrary code execution under the context of the application.
**Prevention:** Never use `pickle` for caching data that is stored on the filesystem, especially in world-writable directories. Always use safe serialization formats like `json`, `msgpack`, or `yaml.safe_load` when storing or transmitting serialized data that crosses trust boundaries.
